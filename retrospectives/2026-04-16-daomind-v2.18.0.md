### 关键决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 构造参数 | `(monitor: DaoUniverseMonitor)` | monitor 提供 heatmapEngine，是 agent 健康状态的输出层 |
| 独立注册表 | `new DaoAgentRegistry()` | 不影响 DaoUniverse.agentRegistry 全局状态 |
| 消息总线 | 代理全局 `daoAgentMessenger` | DaoBaseAgent 内部硬绑定全局 messenger，无法注入替换 |
| monitor 集成 | `heatmapEngine.record('ren', type, 'daoAgents', ...)` | 直接调用引擎 API，避免依赖 DaoUniverse.snapshot() 的间接路径 |
| terminate 容错 | try/catch agent.terminate() | Agent 可能已是 deceased 状态，需忽略状态机异常 |

## 3. DaoBaseAgent 消息总线约束

**关键发现**：`DaoBaseAgent.send()` 硬绑定全局 `daoAgentMessenger` 单例：
```typescript
send(to, action, payload): void {
  daoAgentMessenger.send(this.id, to, action, payload);  // 无法注入
}
```

**影响**：无法为 `DaoUniverseAgents` 创建独立的 messenger 隔离层。

**解决方案**：`DaoUniverseAgents.send()` 代理同一全局 `daoAgentMessenger`，
确保与 `agent.send()` 共享消息总线，`history()` 从 `daoNothingVoid` 读取历史，两者语义一致。

---

## 4. monitor 集成模式

`DaoUniverseMonitor.feed()` 读取 `_universe.snapshot()`（全局 DaoUniverse 的注册表），
无法感知 `DaoUniverseAgents` 的独立 `_registry`。

**解决方案**：直接调用 `monitor.heatmapEngine.record(...)` 注入数据：
- `spawn()` 时：注入新 agent 的 agentType 计数（ren 通道）
- `snapshot()` 时：注入当前全量 byType 分布 + errorRate（deceased 比率）

这使得 `monitor.capture()` 的热力图部分能反映 `DaoUniverseAgents` 的状态。
