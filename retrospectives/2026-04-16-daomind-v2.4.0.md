## 第二章：目标背景

### 接续 v2.3.0

v2.3.0 建立了以下基础设施：
- `DaoBaseAgent`：抽象基类，含 `send()`/`onMessage()`/生命周期
- `DaoAgentMessenger`：Agent 间消息总线
- `DaoAgentContainerBridge`：Agent-容器生命周期同步
- `DaoAgentRegistry`：Agent 注册中心

但整个系统缺少"具体 Agent 范例"——用户不知道如何继承 `DaoBaseAgent`。

### 本次目标

实现三个具体 Agent，覆盖三类典型用途：

| Agent | 角色 | 设计典型性 |
|-------|------|-----------|
| `TaskAgent` | 任务执行者 | 内部状态（队列）+ 完成广播 |
| `ObserverAgent` | 系统观察者 | 外部事件监听 + 内存管理 |
| `CoordinatorAgent` | 协调调度者 | 多 Agent 协作 + 消息分发 |

## 第三章：执行过程

### 里程碑一：TaskAgent（`packages/daoAgents/src/agents/task-agent.ts`）

**设计思路**：
- 帛书依据："为而不争，功成而弗居"——完成任务，结果广播出去，自身不滞留
- 优先级队列：线性扫描插入，O(n) 但适合中小规模，代码最清晰
- `run-next` 执行后自动 `this.send('*', 'task:completed', result)` 广播

**接口设计**：
```typescript
actions: enqueue | run-next | run-all | status | clear
types: AgentTask, TaskResult, QueueSnapshot
```

**亮点**：`enqueue` 按 `priority` 降序插入，保证高优先级任务始终先执行；`run-all` 递归调用 `run-next`，代码复用良好。

---

### 里程碑二：ObserverAgent（`packages/daoAgents/src/agents/observer-agent.ts`）

**设计思路**：
- 帛书依据："致虚极也，守静笃也，万物并作，吾以观复"——无为而观
- 核心挑战：`initialize()` 注册监听器，`terminate()` 必须精确移除——不能用 `removeAllListeners()`（会影响其他订阅者）

**关键模式**：
```typescript
// 构造函数中保存绑定引用
this.boundHandler = (event) => { this.observations.push(...); };
// initialize() 注册
daoNothingVoid.on('observed', this.boundHandler);
// terminate() 精确移除
daoNothingVoid.removeListener('observed', this.boundHandler);
```

**接口设计**：
```typescript
actions: get-snapshot | get-history | get-by-type | clear
types: Observation, SystemSnapshot
```

快照自动分类三类事件：lifecycle / message / other。

---

### 里程碑三：CoordinatorAgent（`packages/daoAgents/src/agents/coordinator-agent.ts`）

**设计思路**：
- 帛书依据："圣人不积，既以为人己愈有"——自身不执行，服务于整体
- **Roster 模式**：通过 `Set<string>` 维护 Agent ID 名册，不直接持有 Agent 引用，避免内存持有和生命周期耦合
- 所有消息通过 `daoAgentMessenger.send()` 发送，而非直接调用 Agent 方法

**接口设计**：
```typescript
actions: add-agent | remove-agent | assign | broadcast | get-roster | get-assignments | find-agent
types: AssignmentRecord, CoordinatorSnapshot
```

`find-agent` 结合 `daoAgentRegistry.findByCapability()` 在名册范围内按能力查找。

### 问题处理

#### 问题：MessageHandler 返回类型冲突

**错误**：
```
TS2322: Type 'number' is not assignable to type 'void | Promise<void>'
```

**复现**：测试代码 `agent.onMessage((m) => receivedA.push(m))`
`Array.push()` 返回 `number`，但 `MessageHandler` 类型声明返回 `void | Promise<void>`。

**修复**：将箭头函数改为块体形式：
```typescript
// ❌ 隐式返回 number
agent.onMessage((m) => received.push(m));

// ✅ 块体形式，隐式 void
agent.onMessage((m) => { received.push(m); });
```

**教训**：TypeScript 严格模式下，`(x) => array.push(x)` 的返回值类型不自动被忽略，写回调时应养成块体习惯。

## 第四章：关键决策

### 决策 1：优先级队列选线性插入而非堆

**背景**：任务队列需要按 `priority` 降序排列。
**选择**：线性扫描找插入点（`Array.splice(idx, 0, item)`）
**理由**：预期队列长度小于 100，线性方案代码最清晰，可读性优先；若需高性能可后续替换为堆。
**结果**：代码直观，测试充分，正确。

### 决策 2：ObserverAgent 在构造函数绑定 handler

**背景**：EventEmitter 的 `removeListener()` 需要传入同一函数引用。
**选择**：在 `constructor` 中 `this.boundHandler = (e) => { ... }` 并保存。
**理由**：比每次用 `bind()` 更清晰，与 React 类组件的最佳实践一致。
**结果**：测试中 `terminate()` 后确认监听器已移除，无内存泄漏。

### 决策 3：CoordinatorAgent 不持有 Agent 实例引用

**背景**：协调者需要向下属 Agent 发消息。
**可选方案**：A) 直接持有 Agent 实例 Map；B) 只存 ID，通过消息总线发送
**选择**：方案 B（只存 ID + 消息总线）
**理由**：避免协调者与 Agent 生命周期耦合，Agent 终止后不需要特别通知协调者。与 M4 方法论（事件驱动消除耦合）一致。
**结果**：Roster 模式简洁高效。

## 第七章：方法论沉淀

### M6：ObserverAgent boundHandler 模式
凡是需要在生命周期结束时精确移除 EventEmitter 监听器的场景，统一在 `constructor` 中保存绑定引用：
```typescript
class SomeObserver {
  private readonly boundHandler: (event: SomeEvent) => void;
  constructor() {
    this.boundHandler = (e) => this.handleEvent(e);
  }
  start() { emitter.on('event', this.boundHandler); }
  stop()  { emitter.removeListener('event', this.boundHandler); }
}
```

### M7：MessageHandler 回调块体规范
任何 `MessageHandler` / EventEmitter 类型的回调，若内部调用有返回值的方法（push/set/delete 等），一律使用块体 `{ }` 形式：
```typescript
// 统一写法
handler((msg) => { this.store.push(msg); });
```

### M8：Roster 模式（ID 而非实例）
协调者类只存被协调者的 ID（`Set<string>`），通过共享消息总线通信，不持有实例引用。解耦生命周期，降低内存风险。
