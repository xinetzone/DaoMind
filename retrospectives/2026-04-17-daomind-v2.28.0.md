### 阶段二：道衍 AI 对话界面重建

#### 背景

旧「道衍」聊天界面（帛书老子智慧问答 AI）之前被 DaoMind 落地页替换，用户截图显示残存错误「响应中断，请稍后重试」——实际为流式响应中断误判导致，非真实服务故障。

#### 根本问题

旧实现的「响应中断」错误来源于：
1. **AbortError 未区分**：用户主动停止时抛出 `AbortError` 被错误地显示为服务失败
2. **流错误事件未捕获**：SSE 流中间的 `event: error` 事件未被专门处理
3. **非 2xx 响应的 SSE 错误体解析缺失**：`onopen` 阶段对 `text/event-stream` 错误响应的解析不完整

#### 解决方案：4 层错误防护架构

```
Layer 0: onopen — 非 2xx 响应时解析 SSE/JSON 错误体
Layer 1: onmessage — 捕获流中间 event:error 数据包
Layer 2: onerror — 重抛网络底层错误（阻止 fetchEventSource 自动重试）
Layer 3: catch — 过滤 AbortError，只对真实错误展示提示
```

## 关键设计决策

### 为何选择 GLM 5

- 中文语境理解深度强，适合古汉语经典诠释
- 费用低、响应速度快，适合实时问答场景
- 与帛书《道德经》的文言文注释风格高度契合

### 为何不保留落地页

用户明确要求重建聊天界面：「这个项目之前有过聊天界面，被替换成了现在的 DaoMind 落地页，请重新构建」。落地页内容可通过 DaoMind 官方文档站访问，聊天界面的即时交互价值更高。

### 系统提示策略

系统提示置于 Edge Function 中（非前端），原因：
1. 防止用户绕过角色设定
2. 便于迭代系统提示而无需前端发版
3. 系统提示包含所有知识库细节，前端保持简洁

## 经验积累

### 流式 AI 响应的稳健错误处理

```
❌ 错误模式：catch 所有错误 → 总是显示「响应中断」
✅ 正确模式：4层防护 + AbortError 明确过滤
```

关键代码模式：
```typescript
// Layer 3: 只对非 AbortError 显示错误
} catch (err: unknown) {
  const e = err as Error;
  if (e.name !== "AbortError") {
    setError(e.message || "连接失败，请稍后重试。");
  }
}
```

### SSE 中间错误事件的捕获

Edge Function 在流中间返回的错误数据包：
```
event: error
data: {"type":"error","error":{"type":"rate_limit_error","message":"..."}}
```

需要在 `onmessage` 中专门检测 `data.type === "error"`，否则会被忽略，仅在流结束时报告。

### 哲学文本的严谨性

- 所有帛书引用使用「」而非""
- 异文差异明确标注出处章节
- 不创造新的"哲学理论"名词——只引用经典原文
