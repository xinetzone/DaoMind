## 一、本版本做了什么

### 新增：DaoUniverseFacade（packages/daoCollective/src/universe-facade.ts）

v2.25.0 引入了 **全栈自动装配门面**——`DaoUniverseFacade`，
封装了按正确依赖顺序构建全部 17 个 DaoUniverse* 桥接器的逻辑。

```typescript
// 用户只需一行代码：
const dao = DaoUniverseFacade.create('/path/to/project');
await dao.diagnose();
const snap = dao.snapshot();
```

内部自动装配顺序：

```
universe → monitor → clock → feedback → audit(universe, projectRoot) → scheduler →
nexus(monitor, clock) → docs → spaces → qi → skills → pages →
agents → apps → times → modules → benchmark → diagnostic(audit, benchmark)
```

## 二、关键设计决策

### 2.1 Object.freeze + satisfies 双重保证

```typescript
this._stack = Object.freeze({
  universe, monitor, clock, ..., diagnostic,
} satisfies DaoUniverseStack);
```

两个关键词组合的作用：
- **`satisfies DaoUniverseStack`**：TypeScript 编译期验证 `_stack` 字面量包含所有 18 个字段且类型正确，不会丢失具体类型信息（与 `as DaoUniverseStack` 不同）
- **`Object.freeze()`**：运行时防突变——外部代码无法覆写 `_stack.monitor = xxx`

### 2.2 projectRoot 传递链

`DaoUniverseAudit(universe, projectRoot)` — 仅这一个桥接器需要 `projectRoot`。
Facade 的 constructor 参数 `projectRoot = process.cwd()` 默认值与 `DaoUniverseAudit` 保持一致。

### 2.3 DaoFacadeSnapshot 五维设计

```typescript
interface DaoFacadeSnapshot {
  system:     DaoSystemSnapshot;   // 基础宇宙层
  monitor:    MonitorSnapshot;     // 五感引擎
  bench:      BenchmarkSnapshot;   // 性能基准摘要
  qi:         QiSnapshot;          // 混元气总线状态
  diagnostic: DiagnosticSnapshot;  // 综合诊断摘要
}
```

有意**不包含** audit 结果（audit 是异步 FS 扫描，snapshot 应是同步轻量操作）。
如需 audit 数据，用 `facade.diagnose()` 获取完整 `DiagnosticRecord`。

## 三、调试：DaoUniverseNexus 双参数陷阱

**错误**：`packages/daoCollective/src/universe-facade.ts:135:24 - error TS2554: Expected 2 arguments, but got 1.`

**根因**：其他大多数桥接器只需 1 个参数，但 `DaoUniverseNexus(monitor, clock)` 需要 2 个。

```typescript
// 错误（初版）
const nexus = new DaoUniverseNexus(monitor);

// 正确
const nexus = new DaoUniverseNexus(monitor, clock);
```

**教训**：在 Facade 构建序列中，每个新桥接器加入时必须查看其 constructor 签名。
不能假定所有桥接器都是单参数形式。

## 四、测试策略：独立容器 vs 根宇宙容器

**关键认知**：`DaoUniverseApps._container` 与 `DaoUniverse.appContainer` 是独立的 `DaoAppContainer` 实例。

错误的测试写法：
```typescript
facade.apps.register({ id: 'app', ... });
await facade.apps.start('app');
const snap = facade.snapshot();
expect(snap.system.apps.total).toBeGreaterThanOrEqual(1); // ❌ 总是 0
```

正确写法（检查 bridge 自己的 snapshot）：
```typescript
facade.apps.register({ id: 'app', ... });
await facade.apps.start('app');
expect(facade.apps.snapshot().total).toBeGreaterThanOrEqual(1); // ✅
```

这是 DaoUniverse* 架构的设计原则：每个桥接器维护独立实例，不污染全局单例。

## 七、下一步（v2.26.0 预告）

DaoUniverseFacade 是"构建者"——下一步引入"消费者"。

**DaoUniverseHealthBoard**：以 `DaoUniverseFacade` 为输入，提供聚焦的健康视图：
- `check()` — 同步健康检查，从 facade.snapshot() 蒸馏出 `HealthEntry`
- `trend()` — 分析最近 N 次 monitorScore 走势：improving / stable / degrading / unknown
- `history()` — 只读健康记录列表
- `snapshot()` — HealthBoardSnapshot 摘要

这将是首个**纯消费者模式**类——不创建任何新子系统，只读取现有 facade。
