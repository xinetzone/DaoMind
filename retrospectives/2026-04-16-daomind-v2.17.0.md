### 关键决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 构造参数 | `(scheduler: DaoUniverseScheduler)` | 调度器提供时序心跳，是刷新触发的来源 |
| binding updater 连线 | 构造时 setUpdater → tree.update | 零胶水代码，notify → tree.update 自动闭环 |
| scheduleRefresh 覆盖语义 | 旧任务先取消 | 防止同组件多刷新任务叠加 |
| 一次性任务 | 执行后 _refreshTasks.delete | 明确任务生命周期，cancelRefresh 前需检查 |
| unmount 联动 | 卸载时同步取消刷新任务 | 防止已卸载组件的僵尸任务 |
| totalMounted 计数 | DaoComponentTree.traverse() | DaoComponentTree 无 size API，只能 DFS 遍历 |

### 新类型

## 3. DaoComponentTree.traverse() 的根节点语义

**关键发现**：`traverse()` 从 `rootId` 出发（第一个 mount 的组件），
多个独立 mount 的平级组件只有第一棵树被遍历。

**影响**：`totalMounted` 仅统计可达树节点，不包含独立挂载的第二棵树。

**测试调整**：`snapshot()` 测试中改为单棵树场景（mount root + 无独立平级组件），
避免触发此语义导致计数不符合预期。

---

## 4. DaoStateBinding.setUpdater() 的连线模式

`setUpdater((componentId, property, value) => tree.update(componentId, { [property]: value }))` 
这是本版本最优雅的设计点——绑定层与渲染树的解耦由 updater 函数桥接，
`notify(path, value)` 调用后全自动驱动组件 prop 更新，调用方无需任何胶水代码。
