## 3. TianQiChannel 签名 Bug（核心发现）

**现象**：`TianQiChannel.broadcast()` 总是导致 `HunyuanBus.totalDropped++`，从不 emit。

**根因分析**：

```typescript
// TianQiChannel.broadcast() 中（tian-qi.ts 第 39-57 行）：
const headerPayload = JSON.stringify({
  id, type, source, target: undefined,
  priority, ttl,
  timestamp: Date.now(),   // ← 第一次 Date.now()（签名用）
});
const signature = this.bus['signer'].sign(headerPayload, 'root-secret');

const message: DaoMessage = {
  header: {
    id, type, source, target: undefined,
    priority, ttl,
    timestamp: Date.now(),   // ← 第二次 Date.now()（消息用，与签名不同）
    signature,               // ← 附加到 header
    encoding: 'json',        // ← 也不在签名 payload 中
  },
  body,
};
```

HunyuanBus.send() 验证时：
```typescript
const payload = JSON.stringify(message.header);
// = JSON.stringify({ id, type, source, target, priority, ttl, timestamp2, signature, encoding })
// ≠ signed payload（timestamp1, 无 signature 字段, 无 encoding 字段）
```

**三重不匹配**：
1. `timestamp1 ≠ timestamp2`（两次 Date.now() 不同）
2. `signature` 字段本身被纳入验证 payload（循环引用）
3. `encoding` 字段不在签名 payload 中

**DaoUniverseQi 解决方案**：`broadcast()` 直接调用 `bus.send()`，构建无签名消息。无签名 → HunyuanBus 跳过验证步骤 → 消息正常路由。

## 4. DiQiChannel 聚合缓冲机制

**设计**：`DiQiChannel.report()` 使用窗口聚合缓冲（默认 1000ms）：
- 第一次 report → 仅缓冲（不发送到 bus）
- 第二次 report（同 key，在窗口内）→ 更新缓冲
- 窗口过期后的下一次 report → flushEntry（发送旧数据）→ 缓冲新数据

**影响**：`report()` 单次调用不会使 `totalEmitted++`，是预期行为。

## 5. subscribe/unsubscribe 设计

```typescript
subscribe(channelType: QiChannelType, handler): () => void {
  return this._bus.subscribe(channelType, handler);
}
```

`HunyuanBus.subscribe()` 在 EventEmitter 上注册 listener，返回 `removeListener` 包装。
`bus.listenerCount(channelType)` 可验证注册/注销。
