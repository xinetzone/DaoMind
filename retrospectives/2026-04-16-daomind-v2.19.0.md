### 关键决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 构造参数 | `(agents: DaoUniverseAgents)` | agents 提供 send()，是应用事件广播的通道 |
| 独立容器 | `new DaoAppContainer()` | 不影响全局 daoAppContainer 单例 |
| 独立生命周期 | `new DaoLifecycleManager()` | 不影响全局 daoLifecycleManager 单例 |
| 双轨记录 | `_lifecycle.emit()` + `agents.send()` | lifecycle 保存转换历史，agents 广播跨层可查 |
| 应用状态前置读取 | `before = container.get(id)` + `fromState = before.state` | start/stop 前先读取 fromState 供 lifecycle.emit 使用 |

## 3. DaoLifecycleManager.emit() 语义

`DaoLifecycleManager.emit(appId, from, to)` 记录 `{ from, to, timestamp }` 到历史。
`onStateChange(appId, cb)` 注册监听器，emit 时回调触发。

**关键发现**：`onStateChange` 只触发 `emit` 后的事件，不会追溯历史。
`dispose()` 取消订阅后，后续 emit 不再触发。

---

## 4. 依赖链验证

`DaoAppContainer.start()` 内置依赖检查：
```typescript
// start() 检查 dependencies 中的每个 depId 是否处于 running 状态
// 若 depId 未 running → 抛出异常
```
`DaoUniverseApps.start()` 直接透传此异常，无需额外处理。

---

## 5. restart() 的双层语义

`_container.restart(id)` 内部执行 `stop → start`，状态机已处理。
但 `_lifecycle` 和 `agents.send()` 无法感知 container 内部的状态变化，
因此 `DaoUniverseApps.restart()` 在 `_container.restart()` 完成后，
补发 `_lifecycle.emit(id, 'stopped', 'running')` 和 `agents.send('app:restarted')`，
确保历史记录连续。
