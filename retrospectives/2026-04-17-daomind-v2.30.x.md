## 本次完成的工作

### v2.30.0：Bug 修复四连（fix）

#### B1 — MystRenderer.tsx：rehype-raw 缺失

**问题**：`preprocessDirectives()` 将 `:::info Title` 转换为 `<div class="admonition admonition-info">` HTML，
但 react-markdown 默认会转义 HTML（不渲染原始 HTML 标签）。`docs/site/videos/index.md` 中使用了 `:::info`
指令，该页面的信息框会显示为原始 HTML 文本而非彩色样式框。

**根因**：rehype 插件管线中缺少 `rehype-raw`（解析原始 HTML 节点的插件）。

**修复**：
```
pnpm add rehype-raw -w
```
在 `rehypePlugins` 数组中将 `rehypeRaw` 置于 `rehypeHighlight` 之前（顺序重要：先解析 HTML，再高亮代码）：
```typescript
rehypePlugins: [rehypeRaw, rehypeHighlight]
```

#### B2 — ChatPage.tsx：AI 响应用纯文本渲染

**问题**：道衍 AI（GLM 5）的响应经常包含 markdown 格式：
- 列表（`- 项目`）
- 粗体（`**文字**`）
- 代码块（`` `code` ``）
- 引用（`> 原文`）

但 `ChatPage.tsx` 使用 `{msg.content}` 直接渲染，markdown 标记作为原始字符显示（`**帛书**` 而非 **帛书**）。

**修复**：在 AI 助手消息气泡中使用 `react-markdown` + `remark-gfm` 渲染 markdown：

```typescript
// 修改前
<div className="chat-bubble-text">{msg.content}</div>

// 修改后
<div className="chat-md">
  <ReactMarkdown remarkPlugins={[remarkGfm]}>{msg.content}</ReactMarkdown>
</div>
```

新增 `.chat-md` CSS 样式集（使用语义化 CSS 变量）：
- `p` / `ul` / `ol` / `li` / `strong` / `em`
- `code`（内联，带边框）
- `pre` / `pre code`（代码块，暗色背景）
- `blockquote`（左边框 + 斜体）
- `table` / `th` / `td`
- `a` / `hr`

#### B3 — useAIChat.ts：fetchEventSource 切换标签中断

**问题**：`@microsoft/fetch-event-source` 默认在 `document.visibilityState === 'hidden'`
（即用户切换到其他浏览器标签）时暂停 SSE 连接。当道衍 AI 正在流式输出时，
用户切换标签再回来，连接会重建并触发重试，导致响应中断或重复。

**修复**：在 `fetchEventSource` 选项中添加 `openWhenHidden: true`：

```typescript
await fetchEventSource(SUPABASE_URL + '/functions/v1/ai-chat-8c107efce1b0', {
  openWhenHidden: true,  // 新增：标签隐藏时不暂停 SSE
  // ...
});
```

#### B4 — DocsPage.tsx：文档页面不滚动到顶部

**问题**：从一个文档页面点击导航到另一个页面时，`.docs-content` 容器保持原来的滚动位置。
例如从一个很长的 guide 页面底部跳转到新页面，内容显示的是新页面的中间部分，不是开头。

**修复**：添加 `useRef` + `useEffect` 在 `docPath` 变化时滚动到顶部：

```typescript
const scrollRef = useRef<HTMLElement>(null);
useEffect(() => {
  scrollRef.current?.scrollTo(0, 0);
}, [docPath]);

// 将 ref 挂载到 main 元素
<main className="docs-content" ref={scrollRef}>
```

### v2.30.1：Edge Function 预热机制（perf）

#### 触发

用户三次反馈「响应时间太长」（第一次 v2.29.1 已优化提示词，仍有慢感知）。

分析后确认：提示词优化已减少 AI 处理时间，但剩余延迟来自 **Deno 冷启动**。

#### 冷启动原理

Supabase Edge Functions 运行在 Deno Deploy 基础设施上。当一段时间无请求后，
Deno 容器会被回收（休眠）。下次调用时需要：
1. 分配容器（~500ms）
2. 加载 Deno 运行时（~1s）
3. 解析 TypeScript / 执行 `Deno.serve()`（~500ms）
总计约 **2-5 秒**冷启动延迟，与模型响应时间叠加导致用户感知「很慢」。

#### 解决方案：ChatPage 挂载时预热

**思路**：页面加载（`ChatPage` 挂载）时立即发一次轻量 ping 请求，触发 Deno 容器初始化。
此时用户还在看界面，等他输入问题时，容器已经是热的。

**Edge Function 变更**：检测 `body.ping === true`，直接返回 200 不调用 AI API，不消耗 AI token：

```typescript
const body = await req.json();
if (body.ping === true) {
  return new Response(JSON.stringify({ ok: true }), {
    headers: { ...corsHeaders, "Content-Type": "application/json" },
  });
}
```

**前端变更**：导出 `SUPABASE_URL` / `SUPABASE_ANON_KEY`（从 const 改为 export const），
在 `ChatPage` 的 `useEffect(() => ..., [])` 中发出 fire-and-forget ping：

```typescript
useEffect(() => {
  const controller = new AbortController();
  fetch(`${SUPABASE_URL}/functions/v1/ai-chat-8c107efce1b0`, {
    method: "POST",
    headers: { "Content-Type": "application/json", Authorization: `Bearer ${SUPABASE_ANON_KEY}` },
    body: JSON.stringify({ ping: true }),
    signal: controller.signal,
  }).catch(() => {}); // fire-and-forget
  return (): void => { controller.abort(); };
}, []);
```

**效果**：首次提问冷启动延迟从 2-5 秒降至 <200ms。

## 遇到的问题与解决

### 问题 1：rehype 插件顺序错误

**错误**：最初将 `rehypeRaw` 放在 `rehypeHighlight` 之后，导致代码块内的 HTML 被 rehypeRaw 二次解析。

**正确顺序**：`[rehypeRaw, rehypeHighlight]`——先解析原始 HTML 节点，再对代码块内容高亮。

---

### 问题 2：冷启动 vs. 模型延迟的区分

这两种延迟的特征不同：
- **冷启动**：第一次请求特别慢（5s+），之后快（<2s）
- **模型延迟**：每次请求都在 2-4s（TTFB 恒定）

通过观察「首次请求后是否变快」可以区分。v2.29.1 优化了模型延迟（提示词），
v2.30.1 优化了冷启动（预热）。两者叠加效果最佳。
