## 2. 核心设计

### 架构层次（执行后）

```
DaoUniverse
  ├── DaoUniverseMonitor (v2.8.0)
  │       ├── DaoUniverseClock (v2.9.0)
  │       │       ├── DaoUniverseFeedback (v2.10.0)
  │       │       └── DaoUniverseScheduler (v2.12.0)
  │       │               └── DaoUniverseSkills (v2.13.0)
  │       └── DaoUniverseNexus (v2.14.0) ← 服务网格 × 宇宙健康
  └── DaoUniverseAudit (v2.11.0)
```

### 关键决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 构造参数 | `(monitor, clock)` | Monitor 提供健康分数，Clock 提供心跳订阅 |
| 服务注册中心 | `new DaoServiceDiscovery()` | 独立实例，不污染全局 daoServiceDiscovery 单例 |
| dispatch() | 纯路由层 | 不依赖 DaoConnectionManager，避免连接状态复杂度 |
| syncHealthNow() | 公开测试钩子 | 允许测试直接触发健康快照录制，无需启动 Clock |
| healthHistory | MAX 100 | 防止内存无限增长，使用 shift() 滚动 |

### 新类型

```typescript
NexusHealthRecord: { timestamp, systemHealth, totalServices, healthyServices, totalRequests, successRate }
NexusDispatchResult: { status: 'dispatched'|'no-service'|'no-target', target, latencyMs }
NexusMetrics: { totalRequests, successCount, failureCount, successRate }
```

## 3. 关键实现细节

**dispatch() 路由策略**：优先使用已注册的路由规则（`_router.resolve(path)`），
若无则动态从 `_discovery.discover(serviceName)` 构建临时规则，
最终委托 `_loadBalancer.select()` 做负载均衡。

**_syncHealth() 原子性**：每次 Clock 心跳调用 `monitor.health()`、
`discovery.healthCheck()`、`metrics()` 三个同步方法，无异步风险。

**dispatch no-target 场景**：候选服务存在但 loadBalancer.select() 返回 null，
此时计入 failureCount。实测 round-robin 策略不会出现此情况，
保留分支为 custom strategy 扩展预留。
