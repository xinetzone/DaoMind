## 2. 核心设计

### 架构层次（执行后）

```
DaoUniverse
  ├── DaoUniverseMonitor
  │       └── DaoUniverseBenchmark (v2.23.0) ← 性能基准 × 宇宙健康感知
  └── DaoUniverseAudit
```

### 关键决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 运行器独立性 | `new DaoBenchmarkRunner()`（每次构建）| 不污染全局单例 |
| 健康关联 | `monitor.health()` 前后各一次 | 将性能报告与宇宙健康实时关联 |
| runSuite 不入 history | 仅 runQuick/runAll 追加记录 | 单套件无完整 DaoPerformanceReport |
| clearHistory 机制 | `_history.length = 0` | 清空记录不影响 runner 内部 results |

### DaoUniverseBenchmark 接口

```typescript
class DaoUniverseBenchmark {
  constructor(monitor: DaoUniverseMonitor)
  async runQuick(): Promise<BenchmarkRunRecord>      // 3 套件 + health 关联
  async runAll():   Promise<BenchmarkRunRecord>      // 6 套件 + health 关联
  async runSuite(name: string): Promise<DaoBenchmarkResult>  // 单套件
  generateReport(format?: 'text' | 'json' | 'markdown'): string
  history(): ReadonlyArray<BenchmarkRunRecord>
  clearHistory(): void
  snapshot(): BenchmarkSnapshot
  get monitor(): DaoUniverseMonitor
  get runner(): DaoBenchmarkRunner
}
```

## 3. 技术挑战与解法

### 3.1 TypeScript project reference 编译副作用

**问题**：添加 `{ "path": "../daoBenchmark" }` 到 daoCollective 的 tsconfig references 后，
`tsc --build` 将 `*.d.ts` / `*.js` 编译产物输出到 `daoBenchmark/src/`（而非 `dist/`）。
这些文件被 git 追踪，污染了源码目录。

**根因分析**：
- `tsconfig.base.json` 中 `paths` 将 `@daomind/benchmark` 映射到 `./packages/daoBenchmark/src`
- 当 daoBenchmark 作为被引用项目首次编译时（tsbuildinfo 缺失），TypeScript 在路径解析模式下
  将声明文件输出到 `src/` 而非 `dist/`，这是 composite project reference + paths 交互的已知副作用

**修复方案**：
1. `git rm --cached packages/daoBenchmark/src/**/*.{d.ts,js}` — 从 git 追踪中移除 36 个文件
2. `.gitignore` 新增 `packages/daoBenchmark/src/**/*.{d.ts,d.ts.map,js,js.map}` 规则
3. 确保 daoBenchmark 的 `tsconfig.tsbuildinfo` 在 daoCollective 构建前已存在（运行顺序正确）

**经验**：所有新增 `@daomind/*` 依赖到 daoCollective 时，应预先检查被引用包的 tsbuildinfo
是否存在，避免首次 `tsc --build` 触发副作用输出。

### 3.2 pnpm install 的 peer 警告

`pnpm install` 显示 `@types/react` peer 版本不匹配警告，但不影响构建和测试。
属于已知的 React 19 生态兼容问题，可忽略。
