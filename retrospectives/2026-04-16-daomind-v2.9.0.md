# DaoMind v2.9.0 复盘 — 2026-04-16

## 版本：v2.9.0 — DaoUniverseClock（daoChronos × daoCollective）

---

## 1. 目标

将 `@daomind/chronos` 的时序能力（`DaoChronos.sync()` 定时心跳）集成到 `daoCollective`，
赋予 `DaoUniverseMonitor` 时间维度：每 N ms 自动执行一次 `capture()` 并推入 `history`。

---

## 2. 核心设计

### 架构层次（执行后）

```
DaoUniverse
  └── DaoUniverseMonitor  (v2.8.0 — 五感引擎，snapshot→MonitorSnapshot)
          └── DaoUniverseClock  (v2.9.0 — daoChronos 时序驱动)
```

### `DaoUniverseClock` 关键决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 驱动方式 | `DaoChronos.sync()` | 复用已有时序基础设施，不重复实现 setInterval |
| 私有核心 | `_doTick(point)` | 统一 start（定时）/ tick（手动）两路触发入口 |
| 手动 API | `tick(): MonitorSnapshot` | 无需定时器，测试友好；不依赖 fake timers 的测试更快 |
| 累计不重置 | `ticks` 跨 stop→start 继续递增 | 体现"逝曰远，远曰反"的时间连续性 |
| callback 管理 | `Set<ClockTickCallback>` | O(1) 添加/删除，去重 |

---

## 3. 测试策略

- 7 个**纯同步**构建测试（无 I/O）
- 5 个 start/stop + 幂等测试（仅检查状态，不等实际时间）
- 5 个 `tick()` 手动测试（无 fake timers）
- **4 个 `jest.useFakeTimers()` 测试**：验证 `setInterval` → ticks 递增 / stop → ticks 冻结 / history 积累

### 经验

`jest.useFakeTimers()` 必须在 `makeStack()` 之前调用，否则 `DaoChronos` 构造函数捕获的 `setInterval` 是真实计时器，`advanceTimersByTime` 无效。
