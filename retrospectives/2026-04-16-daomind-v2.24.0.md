## 一、本版本做了什么

### 新增：DaoUniverseDiagnostic（packages/daoCollective/src/universe-diagnostic.ts）

v2.24.0 引入了 **第一个跨切面组合类**——`DaoUniverseDiagnostic`，
将 `DaoUniverseAudit`（哲学轴）与 `DaoUniverseBenchmark`（性能轴）组合为统一诊断入口。

```
DaoUniverseAudit      ──╮
                          ├──▶ DaoUniverseDiagnostic（宇宙综合诊断）
DaoUniverseBenchmark  ──╯
```

核心方法 `diagnose()` 使用 `Promise.all` 并行执行两轴：

```typescript
async diagnose(): Promise<DiagnosticRecord> {
  const [auditReport, benchRecord] = await Promise.all([
    this._audit.audit(),
    this._benchmark.runQuick(),
  ]);
  const record: DiagnosticRecord = {
    timestamp:     Date.now(),
    auditReport,
    benchRecord,
    runtimeHealth: benchRecord.healthAfter,   // 推导，不额外调用 monitor.health()
  };
  this._history.push(record);
  return record;
}
```

## 二、关键设计决策

### 2.1 并行执行（Promise.all）

最初考虑串行执行（audit → bench），但并行更自然：
- `audit.audit()` 做 FS 扫描（耗时 ~100–500ms）
- `benchmark.runQuick()` 做 3 个 CPU/内存套件（耗时 ~200–800ms）
- 并行总耗时 = max(两者) 而非 sum(两者)，减少约 40–60% 等待时间

**结论**：`Promise.all` 是正确选择，两轴无数据依赖关系。

### 2.2 runtimeHealth 推导

`runtimeHealth` 字段直接取 `benchRecord.healthAfter`（即 benchmark 运行结束后的 `monitor.health()`），
而非在 `diagnose()` 结束时额外调用 `this._benchmark.monitor.health()`。

优点：
- 减少一次 monitor 调用
- 语义上更准确（"性能测量期间的宇宙健康"而非"诊断结束时的快照"）
- `DiagnosticRecord` 不需要携带 `_benchmark.monitor` 引用即可访问健康分数

### 2.3 无新包依赖

`DaoUniverseDiagnostic` 只接受两个已有桥接器作为参数，不引入任何新的 `@daomind/*` 包，
标志着架构从"包桥接"进入"桥接器组合"新阶段。

## 三、测试策略：beforeAll 共享

最初每个测试组独立调用 `diagnose()`，导致 FS 扫描 + benchmark 重复执行 5 次以上，
单文件测试耗时约 20 秒。

**优化方案**：使用 `beforeAll` 在文件级共享一次 `diagnose()` 结果：

```typescript
let sharedRecord: DiagnosticRecord;
beforeAll(async () => {
  const { diag } = makeStack();
  sharedRecord = await diag.diagnose();
}, 60_000);
```

同时将需要独立 `diagnose()` 调用的测试（如 history 增长验证）单独放在各自 `describe` 块内，
通过 `beforeAll` 在块内执行一次。

**效果**：总测试耗时从 ~20s 降至 ~8s，FS+benchmark 调用次数从 6 次降至 2 次。

### 关键约束
- `jest.setTimeout(60_000)` 必须置于文件顶层（非 `describe` 内），否则 `beforeAll` 超时
- `beforeAll` 不接受普通函数（非 async），内部 `await` 必须包裹在 async 中

## 四、generateReport markdown 格式

Markdown 报告包含四部分：
1. **摘要卡片**：版本、时间戳、哲学得分、性能套件数、宇宙健康
2. **哲学六维表**：6 个检查项的 pass/fail 状态与得分
3. **性能套件表**：每个套件的最小/最大/平均值（ms 或 MB）
4. **建议列表**：根据哲学失败项和性能异常自动生成
