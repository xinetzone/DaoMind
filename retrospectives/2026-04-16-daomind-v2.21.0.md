## 2. 核心设计

### 架构层次（执行后）

```
DaoUniverse
  ├── DaoUniverseMonitor
  │       ├── DaoUniverseAgents
  │       │       └── DaoUniverseApps
  │       │               ├── DaoUniverseTimes   (v2.20.0)
  │       │               └── DaoUniverseModules (v2.21.0) ← IoC 容器 × Agent 广播
```

### 关键决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 构造参数 | `(apps: DaoUniverseApps)` | apps 提供 agents.send() 接口，供广播 |
| 独立容器 | `new DaoAnythingContainer()` | 不污染全局 `daoContainer` 单例 |
| activate 广播 | `apps.agents.send('daoModules', '*', 'module:activated', { name })` | 所有 Agent 订阅者可感知模块上线 |
| terminate 广播 | `apps.agents.send('daoModules', '*', 'module:terminated', { name })` | 模块下线通知，供 Agent 清理资源 |
| name 唯一键 | 模块以 `name` 为唯一标识（非 id） | DaoAnythingContainer 以 name 索引 |

## 3. resolve() 平台依赖问题

**关键发现**：`DaoAnythingContainer.resolve(name)` 内部使用 `dynamic import(path)`，
这是文件系统依赖的平台特定 API：
- 在真实 Node.js 环境可以加载 `./modules/auth.js`
- 在 Jest 测试环境动态路径解析行为不确定
- 在浏览器 / Deno 环境行为各异

**解决方案**：测试中仅验证两条错误路径：
1. `resolve('not-registered')` → 抛出"未注册"异常
2. `resolve('registered-but-not-active')` → 抛出"未激活"异常

**影响**：resolve 的实际加载逻辑由 `DaoAnythingContainer` 测试覆盖，桥接层不重复测 FS 行为。

---

## 4. DaoLifecycleManager 语义澄清

**关键发现**：在 v2.21.0 开发过程中，确认 `DaoLifecycleManager.emit()` 是"通知"语义而非"执行"语义——
它仅将状态转换事件广播给监听器，不执行实际的状态迁移逻辑。
状态迁移由 `DaoAnythingContainer.activate/deactivate/terminate()` 执行。

## 6. 与 v2.20.0 的对比

## 7. 已知限制

1. **resolve() FS 依赖**：动态 import 在非 Node.js 平台行为差异，桥接层未提供抽象层
2. **suspend 无对称 resume**：deactivate → suspending 是终态之前的状态，无 resume() 方法
3. **模块名大小写**：DaoAnythingContainer 对 name 的大小写敏感性依赖底层实现
