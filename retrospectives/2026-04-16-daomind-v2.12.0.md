# DaoMind v2.12.0 复盘 — 2026-04-16

---

## 1. 目标

将 `@daomind/times` 的 `DaoScheduler` 融入 `DaoCollective`，
使 DaoUniverse 具备时序驱动任务调度能力：
"待时而动"（系辞传）——任务注册后，由 Clock 心跳触发到期执行。

---

## 2. 核心设计

### 架构层次（执行后）

```
DaoUniverse
  ├── DaoUniverseMonitor (v2.8.0)
  │       └── DaoUniverseClock (v2.9.0)
  │               ├── DaoUniverseFeedback (v2.10.0)
  │               └── DaoUniverseScheduler (v2.12.0) ← 时序驱动调度
  └── DaoUniverseAudit (v2.11.0)
```

### `DaoUniverseScheduler` 关键决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 构造参数 | `DaoUniverseClock` | 与 DaoUniverseFeedback 平级，同挂 Clock |
| DaoScheduler 实例 | `new DaoScheduler()` | 独立实例，不污染全局 `daoScheduler` |
| flush() 循环守卫 | `while (pending() > 0)` | 避免 `next()` 无任务时无限等待 |
| 任务 ID 捕获 | `let capturedId = ''` 闭包 | `DaoScheduler` 自生成 id，handler 异步调用时 id 已赋值 |
| 错误处理 | `flush()` catch 静默 | handler 内已记录 error 状态，flush 继续执行剩余任务 |

### ExecutionRecord 类型

```typescript
interface ExecutionRecord {
  readonly taskId: string;
  readonly executedAt: number;
  readonly status: 'success' | 'error';
}
```

---

## 3. 关键修复

**flush() 错误传播问题**：初版 `flush()` 未捕获异常，任务抛错后循环中断，后续任务被跳过。
修复：在 `while` 循环内添加 `try/catch`，静默吸收来自 `_scheduler.next()` 的抛错，`count++` 仍执行。
